CI框架目录学习

发表于 2017-03-01
  • note：之前不知道 CI框架 的目录的规划依据，目录内的各个文件的封装理由。现在从现有的 CI(3.1.0) 目录倒推，阅读官网介绍以及网上查询的资料，有了一些想法，记录下来。

根目录 —— app 与 sys 简介

首先，对一个新框架，可以从框架的目录来大致了解框架的结构。CI 框架根目录中存在 application，system 两个目录以及其他文件。

从目录来看，application 目录内存在着数个与 system 内的目录相同，但是内容为几乎空的子目录：如 core，helpers，language，libraries。因为第一次接触框架，所以不明白这些目录的意义。谷歌之后，知道 application 目录是应用目录，system 是框架的系统目录，即属于框架本身相关的代码基本都存在于 system 目录内。而 application 目录中的代码是我们自己的项目存放代码的地方，所以我们的基本操作都是在应用目录内。而两个目录内相同的几个目录，是我们用来存放对框架的一些类的扩展。比如系统目录内的类库中没有我们需要的类或者不符合我们需求的类，那么我们可以在应用目录中扩展或者添加需要的类，而不用更改核心目录中的类代码。所以应用目录与系统目录中不同的目录也是根据两个目录的定位来划分的。比如在应用目录中有 MVC 的目录，配置目录，缓存目录以及日志目录。这些是属于项目本身的属性，所以不会存在于系统目录。

system 目录分析

从系统目录 system 中可以看到里面的目录划分。有核心 core 目录，数据库类 database 目录，字体 fonts 目录，辅助函数 helpers 目录，语言 language 目录，类库 libraries 目录。这些目录从名字上就可以很好的明白他们的定位。除开字体，数据库以及语言这三类区别很明显的目录，core 目录，helpers 目录以及 libraries 目录中的文件在一开始让我比较迷糊。

只在三个目录中区分的话，最明显的是 helpers 目录，最简单的分类理由是它们都是函数而不是类。但是为什么它们是函数，而不是类呢？如果将他们封装成类，那么是否属于类库呢？其他两个目录中的文件又是基于什么原因来将其中一类归为 core，而另一类只是类库 libraries 呢？如果不改动的话，那把两个目录合并有什么影响呢？

关于辅助函数为什么不是辅助类，我想应该是函数与类的区别，即这里可以直接用函数来封装一个操作过程，完全用不到面向对象的特性。而关于 core 目录与 libraries 目录。我在 官网上，看到它的介绍，让我获得了启发:

CodeIgniter 是一个小巧但功能强大的 PHP 框架，作为一个简单而“优雅”的工具包，它可以为开发者们建立功能完善的 Web 应用程序。

让我从平时做的最多的说起，将输入一个网址之后的流程简单描述为：1.浏览器发送请求，2.服务器处理请求，3.服务器返回响应给浏览器。而框架处于「服务器处理请求」这一步内。所以框架所要做的至少应该包括对输入以及输出的处理以用来衔接前后两个步骤。

输入之后，需要一个 router 类来处理 url。在 url 中有查询字符串或者数据段需要处理，所以需要一个 uri 类来处理他们。处理完毕则会路由到相应的控制器并调用相应的方法。所以需要 controller 类。而在控制器类当中，将会调用模型或者视图，所以需要一个加载类 loader 类用来加载相应的模型对象或者视图文件。加载视图完毕，则输出到浏览器。
以上需要的类包括：输入类 Input，路由类 router 类，URI 类，controller 类，model 类，loader 类，输出类Output。

上面的流程需要运行在一个文件中，在该文件中按一定的顺序加载相应的类对象。所以需要一个主流程文件 codeigniter。而在这个文件中，要加载各个类并实例化，以及除了这些类之外的一些操作，所以这些操作被封装成常用函数文件 common，将在流程文件的开头被加载。每一个项目都会有属于自己的属性，而系统代码与应用代码是分离的，应用项目的相关配置属性在
application 目录中，所以需要加载这些配置属性，需要配置类 config。另外 CSRF，xss 等一些安全问题也需要保障，所以需要一个安全类 security。而运行项目时，需要对异常进行处理，所以需要一个异常类 exceptions。项目运行的一些必要的信息需要记录，所以需要 Log 类。字符集设定，UTF8 类。因为这是个框架，所以对框架设定的流程，有些人需要定制，所以存在 hooks 类，在流程当中相应位置可以调用对应的钩子类。语言也是必要的，所以有 lang 类。还有可能需要的计时功能，benchmark 类。
上面提到了 CI 的宣传点就是小巧和功能强大。所以上面提到的类：输入类 Input 类，路由类 router 类，URI 类，controller 类，model 类，loader 类，输出类 output 类，codeigniter 文件，common 文件，配置类 config 类，安全类
security 类，异常类 exceptions，日志类 log 类，字符类 utf8 类，钩子类 hooks 类，语言类 lang 类，计时类
benchmark 类总共17个文件是核心，是必不可少的。

回答上面的问题：辅助函数目录里是一些简单的过程操作函数。核心类里是网站流程的必要文件。而类库目录里的类文件就是一些常用的类，被封装在其中。

所以核心类能够完成一个最基本的静态网站的最基本流程了（helpers 目录以及 libraries 目录是让建站更方便）。如果需要变成动态网站，则加入数据库，即 database 目录。如果需要可以更换的字体，则加上字体，即 fonts 目录。如果要国际化，则可以加入语言包，更换语言，即 language 目录。
