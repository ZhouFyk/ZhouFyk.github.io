Config 类：

This class contains functions that enable config files to be managed.

从 config 类的总注释上，可以大致意会到这个类的相关信息，比如属性是配置信息相关的存储变量，方法则用来操作配置信息。

属性

看看属性，只有 3 个，十分简单直观。$config 保存加载的配置参数，$is_loaded 记录已经加载过了的配置文件，$_config_path 保存配置文件所在的路径。

构造方法

然后是构造方法。从代码大致看来，第一步使用 & get_config() 函数获取所有的配置信息，存入 $config。然后对配置信息中 base_url 参数进行处理，不存在则设置一个。最后记录一下日志信息，表明配置类已经实例化了。

细节处，只有处理 base_url 参数时花费了一些篇幅。这部分涉及到 预定义变量 $SERVER 三个配置参数:

  • 'SERVER_ADDR'：当前运行脚本所在的服务器的 IP 地址。

  • 'SCRIPT_NAME'：包含当前脚本的路径。这在页面需要指向自己时非常有用。__FILE__ 常量包含当前脚本(例如包含文件)的完整路径和文件名。

  • 'SCRIPT_FILENAME'：当前执行脚本的绝对路径。

    Note:
    如果在命令行界面（Command Line Interface, CLI）使用相对路径执行脚本，例如 file.php 或 ../file.php，那么 $_SERVER['SCRIPT_FILENAME'] 将包含用户指定的相对路径。

嵌套起来可能无法快速的明白，一个一个来看看。最内层是 basename($_SERVER['SCRIPT_FILENAME'])。basename() 的作用是返回文件名，比如 /etc/passwd => passwd,$_SERVER['SCRIPT_FILENAME'] 是当前脚本的绝对路径。那么这个表达式的返回值就是当前脚本的名称了。我们假设这里获得的值为 index.php，那么 strpos($_SERVER['SCRIPT_NAME'], basename($_SERVER['SCRIPT_FILENAME'])) 就是 strpos($_SERVER['SCRIPT_NAME'], 'index.php')。

外面这层strpos() 是找出查找字符串首次出现的位置。$_SERVER['SCRIPT_NAME'] 是包含当前脚本的路径，比如整个字符串为 http://www.hello.com/path/to/index.php，那么该值就是 /path/to/index.php。所以这个语句的值应该是返回字符串中当前脚本的位置索引。我们假设这个值为 6。那么 substr($_SERVER['SCRIPT_NAME'], 0, strpos(...)) 为 substr($_SERVER['SCRIPT_NAME'], 0, 6)。

最外层呢，substr() 是按规则返回字符串中的子字符串。那么这里呢就是返回 $_SERVER['SCRIPT_NAME'] 这个字符串中，从第一位即索引为 0 的字符开始，总共 6 个字符的字符串。

处理好了 $base_url，然后是 $uri。依旧先判空，为空就不用干了，直接返回值。如果存在就调用 _uri_string($uri) 处理一下 $uri。这个 _uri_string() 方法呢，就是返回对应模式下的 uri。等下看这个方法时再具体说明。

然后呢，也是根据配置中的 uri 的模式来处理 uri。如果 $this->item('enable_query_strings') === FALSE，就看看有没有 $this->config['url_suffix']，有的话就拼接进 $uri 里面去。这里拼接呢，也看看有没有 ?，有的话插入。没有的话直接字符串尾部接上就行。然后返回完整的就行 - return $base_url.$this->slash_item('index_page').$uri;。

当上面那个判断条件不成立，即 $this->item('enable_query_strings') === true 时，看看 ? 存在不存在，不存在要添上去 - $uri = '?'.$uri;。存在自然没问题。最后也是返回正确的拼接了的字符串就行。

总结

config 类的实现十分直观简洁。该类中的属性简单直观。不过方法中除了操作配置参数的方法，还另外有获取 Url 相关的几个方法。在配置类中，在其他方法都是操作配置相关的方法时，存在几个专门操作 Url 的方法，似乎显得有点奇怪。直观上，似乎将这几个方法方在辅助函数里或者 Url 类更合适。当然，CI 的 url_helper.php 文件中确实有这几个函数，但是内部依旧是调用了 config 类对象对应的方法。

仔细想想，这几个方法要操作的 Url 是在配置文件中设置的，所以必须要使用到配置相关。如果不作为内部方法，那还是要加载配置，再进行操作。倒不如放在配置类内部，然后提供一个辅助函数来调用。只能这么接受，是我想不到更好的解决办法。

关于在构造函数内通过 & get_config() 函数获取配置信息。似乎蛮奇怪的，获取配置信息应当也属于配置类的行为吧。看 & get_config() 函数内部除了加载的流程，还有一个批量修改配置的流程，在我看来，这个批量修改的操作也应该属于配置类吧。而当前配置类对配置参数的修改似乎只有 set_item($item, $value) 方法，每次只能修改一个参数。那么综上，为什么不把 & get_config() 函数拆分成加载和批量修改两个方法放在配置类的内部呢。

全局搜索了一下，发现使用到 & get_config() 的地方还是蛮多的。如果将函数放在配置类内部，就需要调用对象，似乎还是放在外面更方便。不过要这么说的话，可以像处理 Url 的相关方法一样，在外部简单封装，内部依旧调用对象使用方法啊。

如果目前这样是最合理的话，那么一定有这么做的理由。除了代码的实现，可能和整个应用的流程相关。查看 CodeIgniter.php，可以看到 config 类实例化的注释：

Instantiate the config class

Note: It is important that Config is loaded first as most other classes depend on it either directly or bydepending on another class that uses it.

而在该类实例化之前，已经有对配置进行操作了。比如 get_config()、config_item()等。那么把配置类的实例化提前到最开始如何呢。之后就调用其相关方法即可。如果要符合目前的设计的话，我想的是，即使将实例化提前，但是同样是配置相关的操作，实例化比直接使用函数多出了实例化对象这一步的资源消耗？

目前只到这里，也许看了其他的类或者方法的实现会对理解这个问题更有帮助。
