ci - core - codeigniter.php

此脚本是框架系统的初始化文件。在执行请求前，对应用加载好所有基础类，最后执行请求。

文件开头对常量 BASEPATH 的判断。该常量在整个框架的唯一入口 index.php 中被定义。所以 ci 在其余每个文件中第一行都对该常量的判断，如果该常量未被定义，则说明是非法进入，直接结束脚本。

  1. 框架版本号。定义一个常量的当前 CI 版本号。

  2. 加载常量。先加载环境常量，然后加载全局常量。

  3. 加载全局函数。

  4. 设置错误异常处理函数用来记录 PHP 错误信息。

  5. 设置子类的前缀。CI 允许用户扩展它的类库，这里设置的前缀就是 CI 对扩展类名的一个命名规范。

  6. 根据配置信息判断是否使用 composer autoload。

  7. 计时。实例化 benchmark 类，设定时间结点，应用执行开始，以及基础类加载开始。

  8. 自定义流程。实例化 Hooks 类。尝试调用钩子，在系统启动前调用用户自定义的操作。钩子 - 扩展框架核心

  9. 加载配置。实例化 config 类，加载更新配置信息。

  10. 设置字符集。检测 mbstring 与 iconv 等扩展是否已经加载，设置对应的标识符和进行相关的操作。

  11. 加载兼容性相关的扩展。解决由于版本不同导致的不兼容问题，自定义一些相关函数。

  12. 实例化 Utf8 类。

  13. 实例化 URI 类。

  14. 实例化 Router 类。

  15. 实例化 Output 类。

  16. 缓存处理。如果已经缓存过相应的内容，则直接返回缓存。

  17. 实例化 Security 类。

  18. 实例化 Input 类。

  19. 实例化 Lang 类。

  20. 加载控制器。包含框架核心控制器类。定义一个方法用来获取核心控制器的单例。如果用户替换了核心控制器，将自定义的控制器也包含。

  21. 计时。时间结点，基础类加载完毕。

  22. 404 检测。由注释可知，到这一步，请求已经被 Router 类检测过，只剩下三个问题：

    1. 类名不存在。
    2. 执行字符串没能通过 file_exist() 的检测。
    3. 请求不存在的页面。

    以上问题都会作为 404 错误处理。除此之外，不论方法是来自控制器还是加载类，都不可以通过 uri 来调用，以及被控制器中以下划线开头的方法调用。

    上述翻译自注释，不过翻译的不够好。无法理解可以直接看原文。

    设置一个标识符 $e404 = FALSE。通过 Router 类对象获取请求的控制器类名和方法名。检测类名，如果类名不存在，或者该类文件不存在，$e404 = TRUE。类文件加载之后，如果该类不存在，或者方法名以下划线 _ 开头，或者该方法为属于核心控制器 CI_Controller，则 $e404 = TRUE。因为在 CI 框架中，下划线开头的方法不属于公有，所以不能被外部调用，以及如果该方法存在与 CI_Controller 中，则说明是非法调用，所以这些情况作为 404 处理。通过以上的检测，如果类中存在方法 _remap()，将参数和方法名重新调整，将方法名改为 _remap()，新参数数组中包含原先的方法名和参数。检测类中有没有该方法，不存在的话，同样 $e404 = TRUE。然后还要检测这个类中的方法能否被调用，不能被调用的话，使用 new ReflectionMethod($class, $method) 获取相关信息，检测是否是公有的，是否是构造函数，如果是，设置 $e404 = TRUE。

    由于代码中这一部分的判断语句较多，纯描述可能不太让人理解的清楚，可以参考代码食用。

    然后，经过以上一系列的处理，$e404 的值也确定了，那么如果 $e404 为 true，接下来进行错误处理了。如果设置了 404 重写，通过 sscanf() 从配置 routes['404_override'] 中获取设置的 $error_class、$error_method。是否正确获取到这两个值，通过该函数的返回值是否为 2 来判断。当不为 2，也就是没有正确获取到需要的值，就将 $error_method 设为 index。到这里就是将 $error_method 设置好了。

    接下来要处理 $error_class。处理一个类的流程已经很熟悉了，首先判断类是否存在，不存在则检测类文件是否存在，存在则包含进来，然后再次检测类是否存在。这里将类是否存在的结果保存在 $e404 中：$e404 = ! class_exists($error_class, FALSE)。以上的每个判断差不多都有另一个分支，不过相关的处理都差不多。

    类的处理过程中，有对 $e404 的操作。所以要对 $e404 再次检测，如果为 false，则整理参数，将错误类和调用的方法保存在一个数组中。否则为 true，即之前的处理都没有改变过它的值，就直接使用 show_404($RTR->directory.$class.'/'.$method)。

    如果方法不需要重新映射，即 $method !== '_remap'，也就是没有使用另外的方法来处理错误，就从之前保存的数组中把参数取出另存在一个变量中。

  23. 调用钩子。钩子节点为 「控制器前」。

  24. 实例化类。设下一个时间结点为 「控制器执行开始」。实例化请求的控制器类。

  25. 调用钩子。钩子节点为 「post_controller_constructor」。

  26. 调用请求方法。源代码：call_user_func_array(array(&$CI, $method), $params)，使用 call_user_func_array() 方法调用请求的类的对象。方法原型参数是不使用引用的，这里加引用的话，嗯，这个问题在 PHP general question 提到过，不过没有准确的答案。我个人的想法是：最直观的，我们对一个函数的参数加上 & 就是想要使用引用，这样在该函数内对该变量所做的改变都会被记录下来。在这里应当也是如此。
