ci - core - controller

CI_Controller 类的注释为：

应用控制器类

作为框架中所有类库对象都会被赋值保存在此类对象中的超类

该类内容极少，只有一个静态私有属性用来保存该类的单例对象。当然作为单例模式的话，也有一个公开的获取该类对象的静态方法 &get_instance()。最后，只有一个构造函数。

构造函数中，首先实例化本类，然后保存到静态私有属性中。接下来派上用场的是两个全局函数：is_loaded()、load_class()。囧。is_loaded() 这个函数在这里也用到了，那我在写 common 的时候就写错了，当时以为这个函数只在 load_class() 内部调用。所以这里通过 foreach 依次从 is_loaded() 中在整个应用流程控制器实例化之前就获得已经实例化的各种类的名称，然后使用 & load_class($class)，将这些类通过引用赋值到控制器对象的变量中，这样在之后使用的时候，就可以直接使用 $this->class_name 这样来调用类对象。

基础核心类对象引用完毕之后，控制器类对象还不能使用。因为除了这些核心类对象之外，不可能完全不使用其他类，如果要使用其他类，那么自然需要一种方式去加载这个类。以前使用魔术方法 __autoload()，后来 php 有了 spl_autoload_register()。手册中为 类的自动加载。不过现在许多框架间有个 PSR 标准，其中 PSR-4 为自动加载规范,使用了命名空间等特性。

不过在 CI 框架中，使用的是一个 loader 类来加载类。所以在控制器构造函数中，核心类之后，就将 loader 类的对象也保存在控制器类对象的属性 load 中，然后对该对象初始化，做好对象内部的准备工作。这样在控制器对象中可以通过 $this->load->class_name 来加载类了。

最后记录下日志 - 控制器类已经实例化。

总结

可以很明显的看到控制器超类使用的单例的思想，因为每次请求响应的整个流程只需要一个控制器对象。不过不知道是不是因为要继承的原因，所以和以往的单例模式的实际写法不相同。我对一个类的单例的写法，通常是一个私有的静态属性 $instance 来保存本类对象，然后构造函数以及魔术方法 __clone() 是私有的。然后这个类通过一个静态公有方法 &getinstance() 来获取该类对象。

所以 CI 中的控制器类从代码来讲似乎并不严格算是单例，因为构造函数是公开的，可以随意 new 对象。但是从使用方法来说，应该是单例的的使用方法：只实例化一个对象，并只使用这一个对象。在 CI 框架中，整个的应用流程中，控制器的实例化只有在 Codeigniter.php 中末尾的一次。

而且常规的单例模式，似乎静态私有属性即可，获取对象的方法并不需要是按引用返回。此处 CI 框架控制器这里使用的是按引用返回。并不清楚这样做的目的。以及虽然设置了 getInstance()，但是在 Codeigniter.php 中还是通过 new 来实例化控制器的啊。

也许在查看了其他的代码之后可以对这个问题有解答。
