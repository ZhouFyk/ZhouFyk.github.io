指令 ssl_protocols 和 ssl_ciphers 可用于限制连接，只允许使用了高版本 SSL / TLS 加密的链接。默认 nginx 使用 ssl_protocols TLSv1 TLSv1.1 TLSv1.2 和 ssl_ciphers HIGH:!aNULL:!MD5，所以通常不必特别配置。要注意的是这些指令的默认值被修改过数次。

https 服务器优化

SSL 操作会消耗额外的 CPU 资源。在多进程系统中，会运行多个进程 worker processes，该数量不会少于可以用的 CPU 核数。最消耗 CPU 资源的操作是 SSL 的握手操作。有两种方式可以最大程度减少每个客户端的握手操作次数：第一种是开启 keepalive 连接，它可以在一个连接中发起多个请求；第二种是重用 SSL 会话参数，以避免并行和后续连接的 SSL 握手。会话信息会被存储在一个进程间共享的 SSL 会话缓存中，该缓存由 ssl_session_cache 指令配置。1M 的缓存包含约 4000 个会话。缓存默认的过期时间是 5 分钟。该值可由 ssl_session_timeout 指令修改。以下是一个 10M 共享会话缓存的多核系统的简单优化配置：

   ssl_session_cache   shared:SSL:10m;
ssl_session_timeout 10m;

基于域名的 https 服务器

当配置多个 https 服务器监听同一个 ip 时，存在一个常见的问题：

       listen          443 ssl;
    server_name     www.example.com;
    ssl_certificate www.example.com.crt;
    ...
}

使用此配置，浏览器会接收到默认服务器的证书，如 www.example.com 而不是请求的域名。这是由 ssl 协议导致的。ssl 链接比浏览器发送 http 请求更早，所以 nginx 不知道请求的服务器是哪个。因此，只会提供默认服务器的证书。

多个名字的 ssl 证书

还有其他方法可以让数个 https 服务器共用一个 ip 地址。但是，这些方法都有其缺点。一个方式是在使用证书时，在 SubjectAltName 的证书字段写数个域名，如 www.example.com 和 www.example.org。但是这个字段的长度是限定的。

另一个方法是在证书中使用通配符域名，如 *.example.org。一个通配符域名能够匹配指定域名的所有的子域名，但是只有一个层级。这个证书只匹配 www.example.org，但是不能匹配 example.com 和 www.sub.example.org。

这两个方法可以结合一下。证书的 SubjectAltName 字段可以包括精确的域名和通配符域名，如 example.org 和 *.example.org。

域名标识

一个在单个 ip 地址上运行多个 https 服务器的更常用的解决方法是 TLS Server Name Indication extension (SNI,RFC 6066)，该扩展允许浏览器在进行 SSL 握手时，传送一个带域名的请求，因此，服务器知道该连接使用哪个证书。然而，支持 SNI 的浏览器十分有限。目前支持的浏览器版本为：

  • Opera 8.0
  • MSIE 7.0 (仅 Windows Vista 或更高版本)
  • Firefox 2.0 和其他使用了 Mozilla 平台 rv:1.8.1 的浏览器
  • Safari 3.2.1 (Windows 版本需要 Vista 或更高版本)
  • Chrome (Windows 版本需要 Vista 或更高版本)

注意：SNI 只允许传输域名，然而，如果请求包含 IP 地址，一些浏览器可能错误地将服务器的 IP 地址作为它的名称传递。所以不要依赖这个方式。

为了在 nginx 中使用 SNI，必需有 nginx 建立的二进制 OpenSSL 库以及运行时动态连接库。OpenSSL 从版本 0.9.8f 起支持 SNI，不过需要在建立它时使用 --enable-tlsext 配置选项。从版本 0.9.8j 起，该选项默认开启。如果 nginx 建立了对 SNI 的支持，那么当使用 nginx 的 -V 命令时会有如下信息：

兼容性

  • 从版本 0.8.21 和 0.7.62 起，可以通过 -V 查看 SNI 的支持状态。
  • listen 指令已从 0.7.14 支持 SSL 参数。之前 0.8.21 只能一起指定 default 参数。
  • SNI 从 0.5.23 起支持。
  • 共享 SSL 会话缓存从 0.5.6 起支持。
  • 版本 1.9.1 起：默认 ssl 协议为 TLSv1,TLSv1.1,TLSv1.2(如果 OpenSSL 库支持)。
  • 版本 0.7.65，0.8.19 起：默认 ssl 协议为 SSLv3, TLSv1, TLSv1.1, and TLSv1.2 (如果 OpenSSL 库支持)。
  • 版本 0.7.65，0.8.19 或之前：默认 ssl 协议为 SSLv2, SSLv3, and TLSv1.
  • 版本 1.0.5 起：默认 ssl 加密为 HIGH:!aNULL:!MD5
  • 版本 0.7.65，0.8.20 起：默认 ssl 加密为 HIGH:!ADH:!MD5
  • 版本 0.8.19：默认 ssl 加密为 ALL:!ADH:RC4+RSA:+HIGH:+MEDIUM
  • 版本 0.7.64，0.8.18 或之前：默认 ssl 加密为 ALL:!ADH:RC4+RSA:+HIGH:+MEDIUM:+LOW:+SSLv2:+EXP
