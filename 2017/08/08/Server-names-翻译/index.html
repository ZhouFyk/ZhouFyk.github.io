Server names 翻译

发表于 2017-08-08

服务器名称 - 域名

Server names 原文链接

  • 翻译中的服务器名称即是域名。

使用指令 server_name 定义的服务器名称决定哪个 server 块用来处理分发的请求。可见 「How nginx processes a request」。server_name 的定义方式有三种，明确的名称，通配符名称，或者正则表达式：

server {
listen 80;
server_name example.org www.example.org;
…
}

server {
listen 80;
server_name *.example.org;
…
}

server {
listen 80;
server_name mail.*;
…
}

server {
listen 80;
server_name ~^(?.+).example.net$;
…
}

当使用 server_name 匹配时，如果匹配到多个结果，比如同时匹配到通配符名称和正则表达式名称，那么会选择第一个匹配到的。下面是优先级：

  1. 明确的名称。
  2. 以星号 * 开头的最长的通配名称，如 *.example.org
  3. 以星号 * 结尾的最长的通配名称，如 mail.*
  4. 第一个匹配到的正则表达式（按配置文件的出现顺序）

同配名称

一个通配名称只会在名称的开始或者结尾出现 *,并且只在边界。如名称 www.*.example.org 和 w*.example.org 都是非法的。但是这些名字可以通过正则表达式指定，如 ~^www\..+\.example\.org& 和 ~^w.*\.example\.org$。一个星号 * 可以匹配数个名称区域。如名称 *.example.org 不只匹配 www.example.com 也匹配 www.sub.example.org。

一种特殊的通配名称 .example.org 可以同时匹配精确名称 example.org 和通配名称 *.example.org。

但是这种写法应该仅限于在以上这种简单的例子中使用，因为数字引用很容易被覆盖重写。

高于 0.6.25 版本的 nginx 支持特殊名称 *，这个符号被错误的解释为匹配一切名称。它并不匹配或者通配所有服务器名称。相反的，它支持的功能是现在 server_name_in_redirect 指令的作用。现在 * 已经被弃用，应该使用 server_name_in_redirect 指令。要注意，并没有什么操作可以通过 server_name 来匹配所有服务器名称或者默认服务器。这里有 listen 指令的属性而不是 server_name 指令。可见 「How nginx processes a request」。不过可以通过监听端口 *：80 和 *：8080，并且指定一个 *：8080 为默认服务器，另一个默认服务器为 *：80：

优化

精确的名称和以 * 开头 / 结尾的通配名称会被存储在三个绑定在监听端口上的散列表内。散列表的大小在配置阶段进行了优化，所以可以使用最高的 CPU 缓存命中来找到名称。 具体设置散列表的细节可见文章 「Setting up hashes」。

保存着精确的名称的散列表首先被查找。如果没有找到，那么保存着以 * 开头的通配名称的散列表会被查找。然后是以 * 结尾的散列表。

查找通配名称散列表比查找精确名称散列表的速度要慢的原因是在查找时是按域名区域查找的。注意，特殊通配名称 .example.org 是存储在通配名称散列表内的而不是精确名称散列表。

正则表达式被顺序测试，因此是最慢的方法，并且是不可扩展的。

由于这些原因，应该尽可能使用精确的名称。比如，如果请求最频繁的名称为 example.org 和 www.example.org，那么显式定义它们是最有效率的：

兼容性

  • 版本大于 0.9.4，支持特殊服务器名称 $hostname。
  • 版本大于 0.8.48，默认服务器名称为空字符串。
  • 版本大于 0.8.25，支持正则表达式形式的服务器名称的命名捕获。
  • 版本大于 0.7.40， 支持正则表达式形式的服务器名称的捕获。
  • 版本大于 0.7.12，支持服务器名称为空字符串。
  • 版本大于 0.6.25，支持通配名称和正则表达式名称作为第一服务器名称。
  • 版本大于 0.6.7，支持正则表达式形式服务器名称。
  • 版本大于 0.6.0，支持尾部为 * 的通配名称。
  • 版本大于 0.3.18，支持特殊格式 .example.org。
  • 版本大于 0.1.13，支持开始为 * 的通配名称。
