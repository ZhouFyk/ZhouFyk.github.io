关于容器

发表于 2017-09-28

phalcon 框架 3.2 文档中有个 DI Container。laravel 框架 5.5 文档中有个 Service Container。

我接触的第一个框架是 CodeIgniter，看 CI 框架 3.1.* (忘记哪个版本了，现在是 3.1.6) 的文档时，并没有看到 container(容器) 的说法，只有个超级对象 $CI，所有需要的类库都会实例化保存在该对象的属性中。如果该超级对象还未实例化，就通过一个全局函数获取它。我的理解大概类似于 单例模式：

在应用这个模式时，单例对象的类必须保证只有一个实例存在。许多时候整个系统只需要拥有一个的全局对象，这样有利于我们协调系统整体的行为。比如在某个服务器程序中，该服务器的配置信息存放在一个文件中，这些配置数据由一个单例对象统一读取，然后服务进程中的其他对象再通过这个单例对象获取这些配置信息。

实现单例模式的思路是：一个类能返回对象一个引用(永远是同一个)和一个获得该实例的方法（必须是静态方法，通常使用getInstance这个名称）；当我们调用这个方法时，如果类持有的引用不为空就返回这个引用，如果类保持的引用为空就创建该类的实例并将实例的引用赋予该类保持的引用；同时我们还将该类的构造函数定义为私有方法，这样其他处的代码就无法通过调用该类的构造函数来实例化该类的对象，只有通过该类提供的静态方法来得到该类的唯一实例。

引用的第一段是单例的思想和应用场景举例。第二段是具体实现的思路。

超级对象 $CI 只要在初始化时，将需要的类实例化，保存在属性中，下回使用直接使用这个属性 $CI->classname 就好了。而继承了该类的子类中要使用对应的类，那么使用 $this->classname 即可。而 CI 保证属性中的类对象按引用返回，使用了两个全局函数来控制 - &load_class()/&is_loaded()。前者用来返回要加载的类的对象，如果已经实例化了，就返回保存的对象，后者用来记录已经实例化了的类。

能够很直观的理解这样的做法，效果如引用的第一段中所说：

有利于我们协调系统整体的行为。

当然不可能和单例模式一模一样。我只是说我的理解是该超级对象实现类似单例模式的效果。

后来在 laravel 的文档中见到了 service container / dependency injection(服务容器 / 依赖注入) 这样的词：

The Laravel service container is a powerful tool for managing class dependencies and performing dependency injection. Dependency injection is a fancy phrase that essentially means this: class dependencies are “injected” into the class via the constructor or, in some cases, “setter” methods.

from laravel-document

Laravel 服务容器是用于管理类的依赖和执行依赖注入的工具。依赖注入这个花俏名词实质上是指：类的依赖项通过构造函数，或者某些情况下通过「setter」方法「注入」到类中。

来自 laravel-china 文档译文。

初一见到我很迷惑。可以网上搜索到相关的 「依赖注入」 等解释，也能够理解它的好处。但是总觉得有些别扭。就像数据公式，我知道这个式子的用法，使用场景，也知道肯定是能由一定的基础推导出来，但是心里还是有点别扭，因为不知道实际是如何推导出来的。

今天看到 phalcon 框架文档中的 Dependency Injection / Service Location 这部分内容，才算有了清晰的认识。

我对这部分内容的简单理解就是，内里是上述 CI 超级对象部分的解释，外名叫 container。文档中详细举例，为什么要 「直接实例化要使用的类」 到 「使用容器，通过依赖注入的方式来使用需要的类对象」。

按我的理解，CI 超级对象十分通俗易懂，很接地气。laravel则算是一种抽象到一定高度。所以 CI 对刚开始接触框架的新手很友好，简单粗暴，什么都明明白白的和你说了。如果新手直接上手 laravel 的话，学习成本比较高，但也仅此而已了，根据它自身的描述，也不是为了给新手的，是给有一定经验的开发者使用的，这些开发者能够看到 「服务容器 / 依赖注入」 就能够直接理解这个词背后的意义，以及了解框架关于这部分的思想。

如上所述，三种框架这部分的思想都一样，只是各自实现的方式不相同。其实大多数框架实现的基本功能的思想都是相同的，除了容器，还有路由，输入输出等，而各自框架的特点部分可能有各自独特的地方而思想不一样。不过之前也只是知道，现在有了更具体的体会。

好像偏题了。哈哈哈哈哈哈。

总结

大概是框架的定位不同吧，文档手册的内容的描述方式和详细程度也不相同。总之今天看到 phalcon 容器这部分内容，让我更好的理解了 CI 和 laravel 的容器部分的思想，消除了我之前看到 laravel 容器部分的别扭感。

phalcon 官网上并没有中文版本，搜索引擎出来的中文文档都是之前的旧版本，简单看了一下这部分，差别还是有的。简单记录一下这部分。

由原文的代码注释也可以清晰理解。创建连接的代码被分离出来由一个连接类 Connection 来专门操作。组件 SomeComponent 只需要在使用一个数据库连接之前，通过方法 setConnection(Connection $connection) 从外部获得连接就可以了。

但是，考虑一下，如果我们在一个应用中，要多次使用到组件，那么就要在使用组件前，多次实例化 Connection 类来创建连接，然后传给 SomeComponent 组件来使用。可以很直观的知道，这很麻烦，也是在浪费资源。 「让我想起了高中的时候总是三天两头要交一个个人信息的表格，因为学校各个要用到学生信息的部门的信息不共享，都是各干各的，所以真的烦。」 那么，我们就使用一个全局注册的机制来存储连接的对象，在任何我们想使用连接的时候，直接使用该连接就好了。「就好像学校终于建立了学生信息统一管理处，学校部门要是需要用到学生信息只需要到管理处拿就好了。」：

可以看出，创建连接的部分被单独放到一个注册类 Registry 中了。在需要的时候可以通过 Registry::getConnection() 来获取数据库连接对象。如果有其他需要，在 Registry 类中增加创建新类对象的方法，然后在要使用的类中增加一个方法来接受这个依赖的注入即可。

以上部分，我们可以理解如何通过依赖注入来解决我们的问题。而将依赖作为参数传入而不是在类内硬编码，这样的解耦可以让我们的应用更具可维护性。

但是简单应用是可以的，如果是长期使用上面讲述的方式来执行依赖注入就会有一些麻烦出现。举例来说，如果一个组件有许多依赖，那我们就需要创建很多的对应的方法或者才构造函数中传递许多参数。而且在使用组件前还要实例化大量的类，可以预见大量的重复代码，应用维护起来也很难受：

$di 其实相当于之前的 Registry 类优化版。之前在注册类内硬编码的创建并返回各种对象的方法换成依赖注入的方式。与之前的方式比较，虽然创建实例依旧需要手动创建，但是在组件中使用的时候，只需要传入容器，而不需要一个个的传入所有的依赖。另外，由于容器使用了动态创建方式，而不是像注册类那样将创建对象的代码硬编码在内部，所以可以按需注册，当需要某个服务时，才注册。而对于框架来说，已经做了许多工作，比如要注册的服务都被集中在一个配置文件中，只需要在配置中填入相应的参数即可。将服务注册到容器这个工作就交由框架来完成了。当然开发者也是可以自己手动注册服务的。
